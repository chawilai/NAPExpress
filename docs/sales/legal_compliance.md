# AutoNAP — ข้อกฎหมายและการปฏิบัติตามระเบียบ

เอกสารสำหรับตอบคำถามลูกค้าเรื่อง **"AutoNAP ถูกกฎหมายไหม? เป็นการ hack ระบบ สปสช. หรือเปล่า?"**

**วัตถุประสงค์:** ให้ทีมขายและลูกค้าเข้าใจสถานะทางกฎหมายของ AutoNAP อย่างชัดเจน
**ผู้ใช้:** Sales team, Customer onboarding, Legal advisor reference
**สถานะ:** v1.0 — ยังไม่ได้ผ่าน legal review (แนะนำให้ทนายตรวจก่อนใช้เป็นเอกสารทางการ)

> ⚖️ **Disclaimer:** เอกสารนี้ไม่ใช่คำปรึกษาทางกฎหมาย ลูกค้าควรปรึกษาทนายของตัวเองก่อนใช้งาน AutoNAP เป็นส่วนหนึ่งของ compliance policy

## 🎯 TL;DR (ตอบสั้น)

**AutoNAP ไม่ใช่การ hack** — เป็น **Robotic Process Automation (RPA)** เหมือนระบบอัตโนมัติทั่วไปในองค์กร เช่น ธนาคาร บัญชี ประกัน

### เหตุผลหลัก 7 ข้อ (ปรับปรุง)

1. **Login ผ่าน ThaiID** — ใช้ระบบ authentication อย่างเป็นทางการของรัฐ ผู้ใช้ยืนยันตัวตน + อนุญาตการเข้าใช้ทุกครั้ง
2. **ใช้ credentials ของลูกค้าเอง** — ไม่ได้ hack เข้าระบบใครอื่น (กรณี direct HTTP mode)
3. **ไม่เก็บข้อมูลผู้รับบริการ** ⭐ — ข้อมูลที่ upload มาผ่าน CSV จะไม่ถูกเก็บหลังการประมวลผลเสร็จ
4. **เก็บเฉพาะ PID masked (xxxx1234) + NAP code** — สำหรับ audit เท่านั้น ไม่มี PII อื่น
5. **ทำในสิ่งที่เจ้าหน้าที่ทำได้อยู่แล้ว** — แค่ทำเร็วขึ้น (เหมือน keyboard macro)
6. **ไม่บุกรุกระบบ** — ไม่ใช้ช่องโหว่, ไม่ข้าม security
7. **Rate-limited + polite** — ไม่ทำให้เซิร์ฟเวอร์ สปสช. ล่ม

### 🛡️ Data Minimization by Design

AutoNAP ออกแบบตามหลัก **"process-then-forget"** — ข้อมูลผู้รับบริการ**ไม่ถูกเก็บ**ในระบบ AutoNAP หลังจากประมวลผลเสร็จ:

```
┌──────────────┐    Upload CSV    ┌──────────────┐    Submit    ┌──────────────┐
│   ลูกค้า     │ ───────────────> │   AutoNAP    │ ──────────> │    สปสช.     │
│              │                  │ (in-memory)  │              │   NAP Plus   │
└──────────────┘                  └──────┬───────┘              └──────┬───────┘
                                         │                             │
                                         │  ลบข้อมูลเคส              │  เก็บข้อมูลเคส
                                         │  (ไม่เก็บ)                │  (อย่างเป็นทางการ)
                                         ▼                             ▼
                                  ┌──────────────┐
                                  │ Audit log   │
                                  │ - PID masked │ ← เก็บเฉพาะเพื่อ audit
                                  │ - NAP code  │    (xxxx1234, RR-xxx)
                                  │ - timestamp  │
                                  └──────────────┘
```

**สิ่งที่เก็บ:**
- ✅ PID **masked** (แสดงเฉพาะ 4 ตัวท้าย, เช่น `xxxx1234`)
- ✅ NAP code ที่ได้จาก สปสช. (เช่น `RR-2569-123456`)
- ✅ Timestamp
- ✅ สถานะ success/fail

**สิ่งที่ไม่เก็บ:**
- ❌ PID เต็ม (13 หลัก)
- ❌ ชื่อ-นามสกุล
- ❌ เบอร์โทร / email / ที่อยู่
- ❌ ข้อมูลทางการแพทย์
- ❌ รายละเอียดการให้บริการ
- ❌ CSV ไฟล์ที่ upload (ลบหลังประมวลผล)

## 1. AutoNAP คืออะไร (ในมุมกฎหมาย)

**คำจำกัดความทางเทคนิค:** Robotic Process Automation (RPA) service สำหรับการบันทึกข้อมูลอัตโนมัติเข้าระบบ NHSO NAP Plus

**ไม่ใช่:**
- ❌ ระบบ hack / crack
- ❌ ระบบเจาะเซิร์ฟเวอร์
- ❌ ระบบปลอมแปลงข้อมูล
- ❌ Scraping ข้อมูลลับของคนอื่น
- ❌ Bypass การ authenticate

**ใช่:**
- ✅ บริการอัตโนมัติ (automation) ที่ทำในสิ่งที่เจ้าหน้าที่คลินิกทำได้อยู่แล้ว
- ✅ ใช้ headless browser (Playwright) ส่ง request แบบ browser ปกติ
- ✅ ใช้ credentials ของลูกค้าที่ลูกค้ายินยอมให้ใช้
- ✅ ส่งข้อมูลที่ลูกค้าเป็นเจ้าของเข้าสู่ระบบที่ลูกค้ามีสิทธิ์เข้าถึงอยู่แล้ว

## 3. กฎหมายไทยที่เกี่ยวข้อง

### 3.1 พ.ร.บ. ว่าด้วยการกระทำความผิดเกี่ยวกับคอมพิวเตอร์ พ.ศ. 2550 (แก้ไข 2560)

**มาตราที่อาจเกี่ยวข้อง + การปฏิบัติตามของ AutoNAP:**

#### มาตรา 5 — เข้าถึงระบบคอมพิวเตอร์โดยไม่ได้รับอนุญาต
> "ผู้ใดเข้าถึงโดยมิชอบซึ่งระบบคอมพิวเตอร์ที่มีมาตรการป้องกันการเข้าถึงโดยเฉพาะ..."

- ✅ **ไม่เข้าข่าย** — AutoNAP ใช้ credentials ที่ลูกค้าให้มาเอง (ซึ่งเป็น credentials ที่ สปสช. ออกให้ลูกค้า)
- ✅ ไม่ได้ข้ามระบบ authentication, ไม่ได้ถอดรหัสผ่าน

#### มาตรา 7 — เข้าถึงข้อมูลคอมพิวเตอร์โดยไม่ได้รับอนุญาต
> "ผู้ใดเข้าถึงโดยมิชอบซึ่งข้อมูลคอมพิวเตอร์..."

- ✅ **ไม่เข้าข่าย** — ข้อมูลที่เข้าถึงคือข้อมูลที่ลูกค้ามีสิทธิ์เข้าถึงอยู่แล้ว (ข้อมูลของผู้รับบริการในคลินิกตัวเอง)

#### มาตรา 9 — แก้ไข/ทำลาย/เพิ่มเติม ข้อมูลคอมพิวเตอร์โดยมิชอบ
> "ผู้ใดทำให้เสียหาย ทำลาย แก้ไข เปลี่ยนแปลง..."

- ✅ **ไม่เข้าข่าย** — AutoNAP เพิ่มข้อมูลในระบบ สปสช. แบบเดียวกับที่ลูกค้าทำมือเอง (ไม่แก้ไข/ลบข้อมูลเดิม, ไม่ปลอมแปลง)

#### มาตรา 10 — ทำให้ระบบคอมพิวเตอร์ทำงานช้าหรือล่ม
> "ผู้ใดกระทำการเพื่อให้ระบบคอมพิวเตอร์ของผู้อื่นทำงานขัดข้อง..."

- ✅ **ไม่เข้าข่าย** — AutoNAP มี rate limiting (หน่วงเวลาระหว่าง request), ไม่ยิง concurrent requests เกินเหตุ

#### มาตรา 14 — นำเข้าข้อมูลเท็จ / ผิดกฎหมาย
- ✅ **ไม่เข้าข่าย** — ข้อมูลที่ส่งเป็นข้อมูลจริงจากการให้บริการ ไม่ใช่ข้อมูลปลอม

### 3.2 พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA)

#### Roles ทางกฎหมาย

| บทบาท | ใคร |
|---|---|
| **Data Controller** (ผู้ควบคุมข้อมูล) | **ลูกค้า** (คลินิก/มูลนิธิ) — ผู้ตัดสินใจว่าจะประมวลผลอย่างไร |
| **Data Processor** (ผู้ประมวลผล) | **AutoNAP** — ประมวลผลตามคำสั่งของลูกค้าเท่านั้น |
| **Data Subject** (เจ้าของข้อมูล) | **ผู้รับบริการ** (ผู้ป่วย/ลูกค้าของคลินิก) |

#### สิ่งที่ AutoNAP ต้องมี (Processor obligations)

1. **Data Processing Agreement (DPA)** — สัญญาการประมวลผลข้อมูลระหว่าง AutoNAP และลูกค้า
2. **Confidentiality** — ไม่เปิดเผยข้อมูลให้บุคคลที่สาม
3. **Security measures** — เข้ารหัสข้อมูล, access log, backup
4. **Deletion on request** — ลบข้อมูลเมื่อลูกค้าร้องขอ
5. **Sub-processor notice** — แจ้งถ้าใช้ผู้ประมวลผลรอง (เช่น VPS provider)
6. **Breach notification** — แจ้งลูกค้าภายใน 72 ชั่วโมงหากเกิดเหตุรั่วไหล

#### สิ่งที่ลูกค้าต้องทำ (Controller obligations)

1. **Consent จากผู้รับบริการ** — PDPA requires consent สำหรับการประมวลผลข้อมูลสุขภาพ (หรือมีเหตุตามกฎหมายอื่น เช่น สัญญา, ภารกิจทางกฎหมาย)
2. **Privacy notice** — แจ้งผู้รับบริการว่าข้อมูลจะถูกส่งไป สปสช. และอาจมีผู้ประมวลผล
3. **Records of Processing Activities** (ROPA) — บันทึกกิจกรรมการประมวลผล

> **💡 สำคัญ:** การส่งข้อมูลเข้า NAP Plus ของ สปสช. เป็น **ภารกิจทางกฎหมาย** (legal obligation) ของคลินิกที่มีสัญญากับ สปสช. อยู่แล้ว → มีฐาน legal basis ชัดเจน AutoNAP เป็นเพียงเครื่องมือที่ช่วยให้ทำภารกิจนี้ได้เร็วขึ้น

### 3.3 พ.ร.บ. หลักประกันสุขภาพแห่งชาติ พ.ศ. 2545

- สปสช. กำหนดให้หน่วยบริการ **ต้องรายงานข้อมูลการให้บริการ** เป็นเงื่อนไขของการเคลมเงิน
- NAP Plus คือระบบรายงานที่ สปสช. จัดให้
- **การบันทึกข้อมูลเข้า NAP Plus ไม่ใช่ทางเลือก — เป็นข้อบังคับ**
- AutoNAP ช่วยให้การปฏิบัติตามข้อบังคับนี้เป็นไปอย่างมีประสิทธิภาพ

**คำถามสำคัญ:** "สปสช. ห้ามใช้ automation ไหม?"
→ จากการตรวจสอบเอกสารสาธารณะ **ไม่พบประกาศของ สปสช. ที่ห้ามการใช้ automation หรือ third-party tools** ในการบันทึก NAP สปสช. กำหนดเพียงว่าข้อมูลต้อง**ถูกต้อง**และ**บันทึกตามกำหนด**

## 4. สิ่งที่ทำให้ AutoNAP "ปลอดภัยทางกฎหมาย"

### 4.1 หลักการออกแบบ 9 ข้อ

1. ✅ **ThaiID Authentication** — Login ผ่าน ThaiID (ระบบยืนยันตัวตนของรัฐ) ผู้ใช้เห็นและอนุญาตทุกครั้ง
2. ✅ **ใช้ credentials ของลูกค้าเท่านั้น** (DirectHTTP mode) — ไม่มีการ share / reuse ระหว่างลูกค้า
3. ✅ **ไม่เก็บ password** — รับ credentials ระหว่าง session เท่านั้น ประมวลผลเสร็จแล้วลบทันที
4. ✅ **Data minimization** ⭐ — ไม่เก็บข้อมูลผู้รับบริการ (PID เต็ม, ชื่อ, เบอร์, ที่อยู่) หลังประมวลผล
5. ✅ **PID masking** ⭐ — เก็บเฉพาะ `xxxx1234` (4 ตัวท้าย) + NAP code สำหรับ audit
6. ✅ **ไม่แก้ไขข้อมูลเดิม** — เพิ่มข้อมูลใหม่เท่านั้น (เหมือนเจ้าหน้าที่ทำเอง)
7. ✅ **Rate limiting** — หน่วงเวลาระหว่าง request ไม่ให้เซิร์ฟเวอร์ สปสช. มีปัญหา
8. ✅ **Data localization** — เซิร์ฟเวอร์อยู่ในไทย (ไม่ส่งข้อมูลออก)
9. ✅ **Right to stop** — ลูกค้ายกเลิกเมื่อไหร่ก็ได้, ข้อมูลที่เก็บไว้ (masked PID + NAP code) ถูกลบตามคำสั่ง

### 4.2 สิ่งที่ AutoNAP **ไม่ทำ** (Never)

- ❌ **ไม่เก็บข้อมูลผู้รับบริการ** (ชื่อ, PID เต็ม, เบอร์, ที่อยู่, ข้อมูลทางการแพทย์)
- ❌ **ไม่เก็บ CSV ไฟล์ที่ upload** — ลบออกจากระบบหลังประมวลผลเสร็จ
- ❌ **ไม่เก็บ password** — ใช้ในระหว่าง session แล้วลบทิ้ง
- ❌ ไม่แก้ไขข้อมูลที่มีอยู่ในระบบ สปสช.
- ❌ ไม่ลบข้อมูลใดในระบบ สปสช.
- ❌ ไม่เข้าถึงข้อมูลของคลินิกอื่น (นอกจากของลูกค้าเอง)
- ❌ ไม่ส่งข้อมูลผู้รับบริการให้บุคคลที่สาม
- ❌ ไม่ขายข้อมูล / ทำการตลาดกับข้อมูลผู้ป่วย
- ❌ ไม่ใช้ข้อมูลเพื่อ train AI model
- ❌ ไม่ยิง DoS / Brute force / Credential stuffing
- ❌ ไม่ bypass CAPTCHA หรือระบบ security ของ สปสช.
- ❌ ไม่หลบเลี่ยง rate limit ที่ สปสช. ตั้งไว้

## 5. เปรียบเทียบกับสิ่งที่ "ผิดกฎหมายแน่ ๆ"

| กิจกรรม | ผิดกฎหมายไหม | เหตุผล | AutoNAP ทำไหม |
|---|---|---|---|
| ใช้ credentials ของคนอื่นโดยไม่ได้รับอนุญาต | ❌ ผิด (ม.5) | Account takeover | ❌ ไม่ |
| ถอดรหัสผ่าน / brute force | ❌ ผิด (ม.5) | Unauthorized access | ❌ ไม่ |
| Scrape ข้อมูลลับของคลินิกอื่น | ❌ ผิด (ม.7) | Unauthorized data access | ❌ ไม่ |
| ปลอมแปลงข้อมูลส่งเข้า NAP | ❌ ผิด (ม.14) + ฉ้อโกง | False claims | ❌ ไม่ |
| ยิง request ให้ระบบล่ม | ❌ ผิด (ม.10) | DoS | ❌ ไม่ |
| เปลี่ยน/ลบข้อมูลในระบบ สปสช. โดยไม่ได้รับอนุญาต | ❌ ผิด (ม.9) | Unauthorized modification | ❌ ไม่ |
| ขายข้อมูลผู้ป่วยให้บริษัทประกัน | ❌ ผิด (PDPA) | Data breach | ❌ ไม่ |
| ส่งข้อมูลที่ลูกค้าเป็นเจ้าของ เข้าระบบด้วย credentials ของลูกค้า | ✅ **ถูกต้อง** | Legitimate use | ✅ **ใช่ (นี่คือ AutoNAP)** |

## 6. ตัวอย่างเปรียบเทียบกับ Automation อื่น ๆ ที่ถูกกฎหมาย

AutoNAP เป็น **RPA (Robotic Process Automation)** — เทคโนโลยีที่ใช้กันในหลายอุตสาหกรรม:

| ตัวอย่าง | วงการ | แบบเดียวกับ AutoNAP |
|---|---|---|
| ธนาคารใช้ RPA เช็คข้อมูลลูกค้าจาก credit bureau | ธนาคาร | ✅ ใช้ credentials ของธนาคาร เข้าระบบ credit bureau อัตโนมัติ |
| บริษัทบัญชีใช้ RPA ดึงใบกำกับภาษีจากสรรพากร | บัญชี | ✅ ใช้ credentials ของลูกค้า เข้าระบบสรรพากร |
| บริษัทประกันใช้ RPA เช็คเครม | ประกัน | ✅ Automation การกรอกฟอร์ม |
| Zapier / Make.com | SaaS ทั่วไป | ✅ Automation ระหว่างระบบ |
| Browser autofill (LastPass, 1Password) | ผู้ใช้ทั่วไป | ✅ กรอกฟอร์มอัตโนมัติด้วย credentials ที่บันทึก |

**ทั้งหมดนี้ถูกกฎหมาย** ตราบเท่าที่:
- ใช้ credentials ของตัวเอง / ได้รับอนุญาต
- ไม่ทำเกินกว่าสิ่งที่มนุษย์ทำได้
- ไม่สร้างความเสียหาย / ขัดขวางระบบ

## 7. FAQ — คำถามที่ลูกค้าถามบ่อย

### Q1: AutoNAP hack ระบบ สปสช. หรือเปล่า?

**A:** ไม่ครับ AutoNAP เป็น **automation (RPA)** เหมือนเทคโนโลยีที่ธนาคาร บริษัทบัญชี ประกันใช้ AutoNAP ทำงานเหมือนเจ้าหน้าที่ของคลินิกกำลังกรอกฟอร์มเอง — แค่ทำเร็วกว่า ใช้ **credentials ของคลินิกเอง** ซึ่ง สปสช. ออกให้

### Q2: สปสช. รู้หรือเปล่าว่าเราใช้ automation?

**A:** จากการตรวจสอบ ไม่พบประกาศของ สปสช. ที่ห้ามใช้ third-party automation สปสช. กำหนดเพียงว่าข้อมูลต้องถูกต้องและทันเวลา AutoNAP **ไม่ได้ซ่อน** ตัวเอง — ใช้ rate limit ปกติ ไม่เล่นกลกับระบบ

**แต่เพื่อความปลอดภัย** ลูกค้าสามารถ:
- แจ้ง สปสช. เป็นลายลักษณ์อักษรได้หากต้องการ (optional)
- ให้ทีม compliance ของตัวเอง review
- ถ้า สปสช. มีคำสั่งห้ามในอนาคต AutoNAP จะเป็นไปตามคำสั่งทันที

### Q3: ถ้า สปสช. ตรวจพบและต้องการยกเลิกบัญชีของเรา?

**A:** เนื่องจาก AutoNAP ใช้ credentials ของลูกค้าและส่งข้อมูล**จริง**เข้าระบบ (ไม่ใช่ข้อมูลเท็จ) **ความเสี่ยงต่ำ** เพราะ สปสช. มักยกเลิกบัญชีเฉพาะกรณี:
- ส่งข้อมูลปลอม (ฉ้อโกง)
- ละเมิด terms of service ชัดเจน
- สร้างความเสียหายกับระบบ

**AutoNAP ไม่ทำอันใดอันหนึ่ง** แต่เราแนะนำให้ลูกค้ามี backup plan (ยังต้องให้เจ้าหน้าที่สามารถบันทึกเองได้หาก AutoNAP มีปัญหา)

### Q4: ข้อมูลผู้ป่วยของเราปลอดภัยไหม?

**A:** ปลอดภัยมากเพราะ AutoNAP **ไม่เก็บข้อมูลผู้ป่วยเลย**:

1. **Process-then-forget** — ข้อมูล CSV ที่พี่ upload มา เราประมวลผลแล้วลบทิ้ง ไม่เก็บ
2. **เก็บแค่ audit trail**:
   - PID แบบ mask (เช่น `xxxx1234` — เฉพาะ 4 ตัวท้าย)
   - NAP code ที่ได้จาก สปสช. (เช่น `RR-2569-123456`)
   - Timestamp + สถานะ (success/fail)
3. **ไม่มีข้อมูลเหล่านี้ในระบบ AutoNAP:**
   - ❌ PID เต็ม 13 หลัก
   - ❌ ชื่อ-นามสกุล
   - ❌ เบอร์โทร
   - ❌ ที่อยู่
   - ❌ ข้อมูลการแพทย์/ผลตรวจ
4. **Login ผ่าน ThaiID** — ไม่ต้องฝาก password ไว้กับเรา
5. **เก็บในไทย** (VPS ไทย ไม่ส่งออก)
6. **ไม่ขายข้อมูล / ไม่ใช้ train AI**
7. **PDPA compliant** — มี Data Processing Agreement (DPA)

**ข้อดีของการไม่เก็บข้อมูล:**
- ถึงระบบ AutoNAP ถูก hack, attacker จะไม่ได้ข้อมูลผู้ป่วย (เพราะไม่มี)
- ลดความเสี่ยง breach notification (PDPA)
- Compliance ง่ายขึ้นมาก — AutoNAP คือ **"data transient processor"** ไม่ใช่ data warehouse

### Q5: ถ้าเกิดปัญหา ใครรับผิดชอบ?

**A:** ขอบเขตความรับผิดชอบกำหนดใน DPA:
- **AutoNAP** รับผิดชอบเรื่อง: ระบบทำงานถูกต้อง, security, PDPA compliance ในฐานะ processor
- **ลูกค้า** รับผิดชอบเรื่อง: ความถูกต้องของข้อมูลที่ส่งมาประมวลผล, consent จากผู้รับบริการ, การใช้งานตามวัตถุประสงค์

สำหรับความเสียหายที่ AutoNAP เป็นเหตุโดยตรง AutoNAP รับผิดตามสัญญา (มักจำกัดที่ค่าบริการ 12 เดือน)

### Q6: เราต้องขอ consent จากผู้ป่วยก่อนใช้ AutoNAP ไหม?

**A:** ลูกค้าควร**แจ้งในใบ consent ที่มีอยู่แล้ว** (หรือ privacy notice ของคลินิก) ว่าข้อมูลอาจถูกประมวลผลโดยผู้ประมวลผลข้อมูลภายนอก (AutoNAP) เพื่อการส่งรายงานเข้าระบบ สปสช.

**ตัวอย่างข้อความเพิ่มใน Privacy Notice:**
> "ข้อมูลการรับบริการของท่านจะถูกบันทึกและส่งเข้าระบบ NAP Plus ของสำนักงานหลักประกันสุขภาพแห่งชาติ (สปสช.) ตามที่กฎหมายกำหนด คลินิกอาจใช้ผู้ประมวลผลข้อมูลภายนอก (AutoNAP) เพื่อช่วยในกระบวนการบันทึกข้อมูลโดยไม่มีการเปิดเผยข้อมูลของท่านต่อบุคคลที่สามอื่น"

### Q7: AutoNAP จะเห็นข้อมูลของผู้รับบริการหมดเลยไหม?

**A:** AutoNAP **เห็นเฉพาะข้อมูลที่จำเป็นสำหรับการบันทึก NAP ชั่วคราว** และ**ลบทิ้งทันทีหลังประมวลผล**:

- **ขณะประมวลผล (in-memory เท่านั้น):** PID, UIC, service date, KP, occupation — ข้อมูลที่ส่ง NAP
- **หลังประมวลผลเสร็จ:** ลบข้อมูลเคสทั้งหมด เก็บเฉพาะ masked PID + NAP code + timestamp

AutoNAP **ไม่เข้าถึง**:
- ❌ Medical records เต็ม
- ❌ Lab results ส่วนตัว
- ❌ การวินิจฉัยที่ละเอียดลึก
- ❌ ประวัติการรักษาในอดีต

ข้อมูลในระบบ AutoNAP หลัง 1 ชั่วโมง = **masked PID + NAP code + timestamp** เท่านั้น

### Q8: ถ้าลูกค้ายกเลิก ข้อมูลจะเป็นยังไง?

**A:** เพราะ AutoNAP **ไม่ได้เก็บข้อมูลผู้ป่วยอยู่แล้ว** คำถามนี้เหลือเฉพาะ audit records:

- **Cancel subscription:** masked PID + NAP code ถูกทำเครื่องหมายลบภายใน 30 วัน
- **Hard delete request:** ลบทั้งหมดภายใน 7 วัน
- **Export (สำหรับ audit):** ลูกค้าขอ audit report ได้ (masked PID + NAP code + timestamp)

**ไม่มีข้อมูลส่วนตัวที่ต้องกังวล** เพราะเราไม่ได้เก็บตั้งแต่แรก

### Q9: ถ้า AutoNAP ไม่เก็บ แล้วจะตรวจสอบย้อนหลังได้ยังไงว่าเคสไหนบันทึกแล้ว?

**A:** เราเก็บ **audit trail** ที่พอเพียงสำหรับการตรวจสอบ:

```
| PID (masked) | NAP code         | Service date | Status  | Timestamp           |
|--------------|------------------|--------------|---------|---------------------|
| xxxx1234     | RR-2569-123456   | 2026-04-10   | success | 2026-04-10 10:30:15 |
| xxxx5678     | VCT-2569-987654  | 2026-04-10   | success | 2026-04-10 10:31:20 |
```

**ใช้ตรวจสอบได้ว่า:**
- เคสไหนบันทึกแล้ว (match ด้วย 4 ตัวท้าย)
- NAP code คืออะไร (ใช้ reconcile กับระบบของคลินิก)
- บันทึกเมื่อไร

**แต่ไม่สามารถ reconstruct ตัวตนได้** — 4 ตัวท้ายของเลขบัตร ปชช. **ไม่ใช่ personal data ตาม PDPA** (ต้องมี context อื่นเพิ่มเพื่อระบุบุคคล)
